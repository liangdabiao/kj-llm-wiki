1. 开篇：为什么你的笔记库总是烂尾？
我们都有过这样的经历：
收藏了 500 篇论文，真正记住的不到 5%

用 ChatGPT 读文档，每次都要重新上传、重新解释背景

Notion/Obsidian 里记了上千条笔记，却越来越乱，最后连自己都懒得搜

问题的本质：传统 RAG（检索增强生成）是"临时查资料"，AI 每次都要从零开始理解你的文档。没有积累，没有链接，没有进化。
图片
Karpathy 在最近的技术分享中提出了一种全新范式：LLM Wiki 让 AI 不是帮你"查"资料，而是帮你"写"一本持续更新的个人百科全书。
2. 核心理念：从"检索"到"编译"
2.1 传统工作流 vs 新范式
传统 RAG 工作流：
你提问 → AI 去海量文档里捞片段 → 拼凑答案 → 下次再问，重新捞一遍
Karpathy 的编译式工作流：
原始资料 → AI "编译"成结构化 Wiki → 形成永久知识图谱 → 你提问 → AI 查阅已整理好的"书本" → 答案归档回 Wiki
关键区别：Wiki 是持久化、复利型的资产。AI 会把每篇新读到的论文整合进现有知识体系，而不是让它们孤立地躺在文件夹里。
"知识库最烦人的不是阅读或思考，而是 bookkeeping（记账式维护），更新交叉引用、标记矛盾、保持摘要新鲜。人类会因为厌烦而放弃，AI 不会。"

 -- Andrej Karpathy

3.1 Raw Sources（原始层）
只读仓库：论文、网页、PDF、图片

工具推荐：Obsidian Web Clipper 一键剪藏网页，配合快捷键自动下载图片到本地

原则：原始资料永不修改，作为"信源基准"

3.2 The Wiki（知识层）
AI 全权管辖：由 LLM 生成和维护的 Markdown 文件集合

包含内容：

📄 实体页：人物、公司、概念专页（像 Wikipedia 条目）

🔗 反向链接：自动标注"这篇论文被哪些主题引用"

⚖️ 矛盾标记：当新论文推翻旧观点时，AI 自动标注冲突

📊 综合看板：研究主题的演进时间线、对比表格

3.3 The Schema（配置层）
AI 的操作手册：比如 CLAUDE.md 或 AGENTS.md

定义规则：目录结构、命名约定、如何处理新资料、如何更新索引

人与 AI 共同进化：随着项目深入，不断调整 Schema 让 AI 更懂你的领域

4. 三大操作：Ingest → Query → Lint
4.1 Ingest（摄取）：不是保存，是"消化"
当你丢给 AI 一篇新论文时，它不会只存个摘要，而是会：
讨论核心要点（与你在聊天窗口确认理解）

撰写/更新实体页（比如"Transformer 架构"的历史版本）

修订相关主题页（如"注意力机制"章节增加新视角）

更新索引和日志

一次摄取可能修改 10-15 个 Wiki 页面，这才是真正的"知识内化"。
4.2 Query（查询）：知识库成为你的对话搭档
当你的 Wiki 积累到 100 篇文章、40 万字时，你可以问复杂问题：
"对比 2025 和 2026 年关于 MoE 架构的三种不同观点"

"生成一个关于我们项目架构决策的 Marp 幻灯片"

"画出我们技术栈依赖关系的 matplotlib 图表"

关键机制：好答案会被回写（file-back）进 Wiki。你的一次深度探索，会成为后续所有对话的上下文基础。
4.3 Lint（清洗）：知识库的"健康体检"
定期让 AI 执行：
扫描矛盾陈述（A 页面说 X 技术好用，B 页面说难用，标记待澄清）

发现孤儿页面（没有任何链接指向的孤立条目）

补全缺失数据（通过网络搜索填补空白）

提出新的研究问题

5. 工具链：Obsidian 作为 AI 的 IDE
Karpathy 的实操配置：
Obsidian：作为可视化前端，查看图谱、浏览关系、阅读幻灯片

LLM Agent（Claude Code / OpenAI Codex / Cursor）：作为后端"写手"，在聊天窗口中执行所有编辑

Marp 插件：将 Wiki 内容一键转为 PPT

Dataview 插件：基于 YAML frontmatter 生成动态表格

Git 版本控制：整个 Wiki 就是一个 Git 仓库，可追溯、可分支

人机分工：
你：负责提出好问题、筛选信源、判断价值

AI：负责交叉引用、摘要更新、格式维护、图谱生成

6. 社区验证：已有开源实现
这篇 Gist 发布仅 3 天，GitHub Stars 已破 5000，社区已涌现出多个实现：
graphify：一个 AI 编码助手技能（支持 Claude Code/Codex 等），可将任意文件夹中的代码、文档、论文及图片自动构建为多模态知识图谱

llm-wiki-kit（MCP 服务器）：让 Claude / Cursor 自主维护知识库

OMEGA：解决大规模 Wiki 的检索问题（本地向量搜索 + 交叉编码器重排序）

sage-wiki：内置 Web UI，无需 Obsidian 也能浏览知识图谱

LLM-wiki：包含内容哈希校验，自动检测原始资料更新后的"过期页面"

8. 结语：你的"记忆外包"时代
Karpathy 将这个模式比作 Vannevar Bush 1945 年提出的 Memex（记忆扩充器），一个私人的、带有关联路径的知识库。Bush 当年没能解决的"谁来维护"问题，现在由 LLM 解决了。
8.1 给读者的行动建议
今天就试试：打开 Obsidian，建一个 raw/ 文件夹和一个wiki/ 文件夹

选一篇你最近读的论文，用 Claude Code 或 Cursor 试试"摄取"流程

写一份简单的 Schema，告诉 AI 你希望如何组织知识

坚持 10 次摄取，你会惊讶于 AI 比你更清楚你"学过什么"

正如评论区一位开发者所说：
"一旦 AI 协议强制执行'每个结论都要回写 Wiki'，知识库就会从完全不同的维度变得更密集。第 7 次会话后，全新的 Claude 会话无需你开口就能回答：'我们上周关于定价公式的决定是什么？'"

这才是真正的第二大脑，不是存储容器，而是会自己生长的思维外骨骼。
